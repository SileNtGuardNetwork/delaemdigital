# Стандарт конверсионного прототипа

Статус: v1 foundation
Назначение: зафиксировать стандарт продающего скелета сайта до дизайна и кода.

## 1. Главная позиция

Конверсионный прототип — это не wireframe и не список секций.

Конверсионный прототип — это логический скелет будущего сайта, который показывает:

- кому продаём;
- что продаём;
- почему это важно;
- как пользователь проходит путь от первого экрана до действия;
- где встроена ИИ-автоворонка;
- какой лид-магнит выдаётся;
- какой трипвайер предлагается;
- какие CTA используются;
- какие данные собираются;
- какие события аналитики фиксируются;
- как сайт готовится к SEO и трафику.

Базовая формула:

```text
Оффер -> Доверие -> Понимание -> Диагностика -> Лид-магнит -> ИИ-автоворонка -> Заявка / Трипвайер -> Аналитика
```

## 2. Что должен выдавать прототип

На выходе должен быть не общий план, а готовый скелет сайта:

```text
1. позиционирование
2. карта аудитории
3. основной оффер
4. структура страниц
5. структура главной
6. логика каждого экрана
7. заголовки и смысловые блоки
8. CTA на каждом этапе
9. лид-магнит
10. вход в Telegram-бот
11. welcome-цепочка
12. трипвайер
13. форма заявки
14. SEO-карта
15. карта событий аналитики
16. mobile order
17. риски и ограничения
```

## 3. Исходные данные

Перед прототипом должны быть готовы:

- диагностика бизнеса;
- архитектура оффера;
- карта ЦА;
- карта боли;
- продуктовая линейка;
- ограничения ниши;
- требования по legal;
- понимание источников трафика;
- желаемое действие пользователя;
- понимание лид-магнита или гипотеза;
- понимание трипвайера или гипотеза.

Если этого нет, прототип получает статус `условно готов`, а не `готов`.

## 4. Главная карта сайта

Прототип должен начинаться с карты:

```text
Источник трафика
-> Первый экран
-> Понимание проблемы
-> Метод / ClientFlow System
-> Оффер
-> Доверие
-> Диагностика / лид-магнит
-> Telegram-бот
-> ИИ-прогрев
-> Трипвайер / заявка
-> Обработка
-> Аналитика
```

## 5. Структура главной страницы

Базовая структура premium ClientFlow-сайта:

### 1. Первый экран

Задача:

- быстро объяснить оффер;
- показать отличие;
- дать главный CTA;
- дать мягкий вход в ИИ-автоворонку.

Обязательные элементы:

- кто говорит;
- для кого;
- что собираем;
- результат;
- механизм;
- primary CTA;
- secondary CTA в лид-магнит / Telegram-бот;
- trust marker.

### 2. Карта проблемы

Задача:

- показать, где бизнес теряет заявки;
- связать боль с ClientFlow System;
- подготовить пользователя к диагностике.

### 3. Метод / ClientFlow System

Задача:

- объяснить, что сайт — часть системы;
- показать маршрут клиента;
- отделить продукт от обычной разработки сайтов.

### 4. Что собираем

Задача:

- показать состав продукта;
- объяснить сайт, ИИ-автоворонку, аналитику, SEO, интеграции.

### 5. ИИ-автоворонка

Задача:

- объяснить Telegram-бота;
- показать лид-магнит;
- показать welcome-цепочку;
- показать ИИ-прогрев;
- показать трипвайер.

### 6. До / после

Задача:

- показать разницу между хаотичным digital и управляемой системой.

### 7. Процесс работы

Задача:

- снять тревогу;
- показать этапы;
- показать контроль качества.

### 8. Продукты / пакеты

Задача:

- дать понятную продуктовую лестницу;
- не превращать всё в дешёвые тарифы.

### 9. Доверие / кейсы / доказательства

Задача:

- доказать подход артефактами;
- показать процесс, не только результат.

### 10. Основатель / команда / экспертиза

Задача:

- показать ответственность;
- объяснить авторскую технологию.

### 11. Лид-магнит / диагностика

Задача:

- дать низкий порог входа;
- перевести пользователя в Telegram-бот;
- запустить ИИ-воронку.

### 12. FAQ / возражения

Задача:

- закрыть возражения по срокам, цене, гарантиям, SEO, трафику, AI, оплате, legal.

### 13. Финальный CTA

Задача:

- дать выбор: заявка, диагностика, Telegram;
- не оставлять пользователя без следующего шага.

## 6. CTA-архитектура

У каждого сайта должно быть три уровня CTA.

### Горячий CTA

Для готового пользователя.

Примеры:

```text
Обсудить проект
Получить разбор системы
Собрать сайт с ИИ-автоворонкой
```

### Мягкий CTA

Для пользователя, который ещё не готов купить.

Примеры:

```text
Пройти диагностику
Получить карту потерь заявок
Получить мини-аудит сайта
```

### Доверительный CTA

Для знакомства.

Примеры:

```text
Посмотреть подход
Перейти в Telegram-канал
Посмотреть кейсы
```

Запрещено делать главным CTA:

```text
Подробнее
Узнать больше
Отправить
Кликнуть
```

## 7. Лид-магнит в прототипе

Каждый прототип должен определить лид-магнит.

Поля:

```text
Название лид-магнита:
Для кого:
Какую боль закрывает:
Что пользователь получает:
В каком формате:
Какие данные собирает:
Как ведёт к трипвайеру:
Как ведёт к основному продукту:
```

Примеры:

- карта потерь заявок;
- мини-аудит сайта;
- scorecard готовности к трафику;
- чек-лист доверия;
- диагностика маршрута клиента;
- персональный PDF-отчёт;
- серия сообщений в Telegram.

## 8. ИИ-автоворонка в прототипе

Прототип обязан описать, как сайт ведёт в ИИ-автоворонку.

Поля:

```text
Точка входа:
Telegram deep-link:
Первое сообщение бота:
Лид-магнит:
Вопросы ИИ-ядра:
Сегменты пользователей:
Welcome-цепочка:
Прогрев:
Трипвайер:
Оплата:
Карточка владельца:
События аналитики:
```

Если ИИ-автоворонка не описана, прототип не готов.

## 9. Трипвайер в прототипе

Каждый сайт с ИИ-автоворонкой должен иметь гипотезу трипвайера.

Поля:

```text
Название:
Цена:
Что входит:
Срок выдачи:
Формат выдачи:
Почему покупают:
Как продаётся в боте:
Что после покупки:
Как ведёт к основному продукту:
```

Примеры:

- экспресс-аудит сайта;
- мини-разбор ClientFlow;
- платный диагностический отчёт;
- шаблон;
- мини-курс;
- платная консультация;
- прототип первого экрана.

## 10. SEO-слой прототипа

Прототип должен включать базовую SEO-карту.

Поля:

```text
Главный поисковый интент:
Основные ключевые фразы:
Title:
Description:
H1:
H2-структура:
FAQ для SEO:
Schema.org:
Страницы под продукты:
Страницы под ниши:
Внутренние ссылки:
```

SEO не добавляется после дизайна. SEO влияет на структуру сайта.

## 11. Карта аналитики прототипа

Прототип должен включать ключевые события.

Минимум:

```text
page_viewed
section_viewed
hero_primary_cta_clicked
hero_secondary_cta_clicked
lead_magnet_requested
bot_started
lead_form_started
lead_form_submitted
tripwire_offered
payment_started
payment_success
owner_lead_card_sent
```

## 12. Mobile order

Прототип должен определить порядок блоков на мобильном.

Правила:

- первый экран не должен быть перегружен;
- CTA должен быть виден рано;
- длинные схемы упрощаются;
- карточки не превращаются в бесконечную простыню;
- форма должна быть удобной;
- бот и лид-магнит должны быть доступны без поиска по странице.

## 13. Copy-скелет

Для каждого экрана прототип должен дать:

```text
Экран:
Роль:
Главный смысл:
Заголовок:
Подзаголовок:
Ключевые тезисы:
CTA:
Возражение, которое закрывает:
Что должно быть видно визуально:
```

Это не финальный copy-doc, но достаточно точный скелет для копирайтинга и дизайна.

## 14. Прототипный Gate

Оценить по 10-балльной шкале.

| Блок | Оценка | Комментарий |
| --- | ---: | --- |
| Ясность оффера |  |  |
| Логика маршрута |  |  |
| CTA-архитектура |  |  |
| Лид-магнит |  |  |
| ИИ-автоворонка |  |  |
| Трипвайер |  |  |
| SEO-логика |  |  |
| Аналитика |  |  |
| Mobile order |  |  |
| Готовность к дизайну |  |  |

Итого: /100

Статусы:

```text
90-100: готов к дизайну
75-89: условно готов, нужны правки
ниже 75: не готов
```

Критичные блокеры:

- нет главного оффера;
- нет CTA;
- нет лид-магнита;
- нет ИИ-автоворонки;
- нет mobile order;
- не понятно, что пользователь должен сделать;
- нет связи с продуктом;
- нет SEO-логики для SEO-проекта.

## 15. Формат итогового прототипа

Каждый прототип должен выпускаться в формате:

```text
1. Краткое позиционирование
2. Основной маршрут пользователя
3. Структура страниц
4. Структура главной
5. Copy-скелет по экранам
6. CTA-карта
7. Лид-магнит
8. ИИ-автоворонка
9. Трипвайер
10. Форма заявки
11. SEO-карта
12. Аналитика
13. Mobile order
14. Риски
15. Gate score
```

## 16. Решение

Фабрика не переходит к дизайну, пока не готов конверсионный прототип.

Правильный порядок:

```text
Диагностика -> Оффер -> Конверсионный прототип -> Copy-doc -> Дизайн -> Код -> Интеграции -> QA
```

Конверсионный прототип — главный мост между бизнесом и сайтом.
